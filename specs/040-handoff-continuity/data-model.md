# Data Model: NSUserActivity / Handoff / Continuity Module

**Feature**: 040-handoff-continuity
**Date**: 2026-04-30

This module ships **four** TypeScript types. None are persisted to disk — all are in-memory or platform-managed.

---

## 1. `ActivityDefinition`

The contract shape used by the composer, the JS bridge, and the native bridge. This is the single in-memory representation of a Handoff activity definition.

```ts
export interface ActivityDefinition {
  /**
   * Reverse-DNS string. MUST appear in Info.plist NSUserActivityTypes
   * for the OS to broadcast it. Default: HANDOFF_DEMO_ACTIVITY_TYPE.
   */
  activityType: string;

  /**
   * User-visible title surfaced on the receiving device's
   * Handoff hint banner / icon badge.
   */
  title: string;

  /**
   * Optional. If set and parseable, becomes the activity's
   * webpageURL — relevant for Universal Links continuation
   * (deferred — see Story 6).
   */
  webpageURL?: string;

  /**
   * Free-form payload. Keys are arbitrary strings; values
   * are strings only at the composer level (the native side
   * accepts plist-safe Any but the educational composer
   * keeps it simple).
   */
  userInfo: Record<string, string>;

  /**
   * Subset of Object.keys(userInfo). The OS may decline to
   * deliver the activity to the receiving device if any
   * required key is missing.
   */
  requiredUserInfoKeys: string[];

  /** Whether the activity is eligible for Handoff (default true). */
  isEligibleForHandoff: boolean;

  /** Whether the activity is eligible for Spotlight search (iOS 9+). */
  isEligibleForSearch: boolean;

  /** Whether the activity is eligible for Siri prediction (iOS 12+). */
  isEligibleForPrediction: boolean;
}
```

**Validation rules** (enforced in `ActivityComposer`, FR-010):

| Field                   | Rule                                                                   |
|-------------------------|------------------------------------------------------------------------|
| `activityType`          | Non-empty; matches `^[a-zA-Z0-9._-]+$`; contains at least one `.`      |
| `title`                 | Non-empty (defaults to `'Handoff demo activity'`)                      |
| `webpageURL`            | Empty OR `new URL(value)` succeeds AND protocol is `http:` or `https:` |
| `userInfo`              | No duplicate keys (KeyValueEditor enforces)                            |
| `requiredUserInfoKeys`  | Every entry is a key in `userInfo`                                     |
| eligibility flags       | Booleans only                                                          |

## 2. `ContinuationEvent`

Payload prepended into the IncomingLog when the AppDelegate forwards an `application(_:continue:restorationHandler:)` event.

```ts
export interface ContinuationEvent {
  activityType: string;
  title: string;
  webpageURL?: string;

  /**
   * Free-form payload from the sending device. Values are
   * `unknown` here (vs `string` on ActivityDefinition) because
   * Apple permits any plist-safe type and we don't want to
   * lose information at the receiver.
   */
  userInfo: Record<string, unknown>;

  /** Sorted, deduplicated; native side guarantees this. */
  requiredUserInfoKeys: string[];

  /**
   * Generated client-side at receipt with new Date().toISOString().
   * NOT generated by the native bridge — the educational point
   * is "this is when the JS hook saw the event".
   */
  receivedAt: string;
}
```

**Discard rule** (FR-015 / Edge Case #4): If `activityType` is missing or not a string, the hook silently discards the event, optionally `console.warn`s in `__DEV__`, and the log is unchanged.

## 3. `HookState`

The slice exposed by `useHandoffActivity()`.

```ts
export interface HookState {
  /** Mirror of the last successfully made-current activity. */
  currentActivity: ActivityDefinition | null;

  /** Most recent first; max length 10 (FR-014). */
  log: ContinuationEvent[];

  /** False on Android / Web; true on iOS. */
  isAvailable: boolean;
}
```

**State transitions**:

| Action                                       | `currentActivity`           | `log`                             |
|----------------------------------------------|-----------------------------|-----------------------------------|
| Mount                                        | `null`                      | `[]`                              |
| `setCurrent(def)` resolves                   | `def`                       | unchanged                         |
| `resignCurrent()` resolves                   | `null`                      | unchanged                         |
| `onContinue` event with valid payload        | unchanged                   | `[event, ...log].slice(0, 10)`    |
| `onContinue` event with missing `activityType`| unchanged                  | unchanged (event discarded)       |
| Unmount                                      | (cleared with component)    | (cleared with component)          |

## 4. `ComposerFormState`

Local form state mirroring `ActivityDefinition` plus per-field validation errors. **Not exported** — internal to `ActivityComposer.tsx`.

```ts
interface ComposerFormState {
  // mirror of ActivityDefinition
  activityType: string;
  title: string;
  webpageURL: string;          // empty string = absent
  userInfoRows: { key: string; value: string }[];  // ordered, KeyValueEditor source
  requiredKeys: Record<string, boolean>;           // key → checked
  isEligibleForHandoff: boolean;
  isEligibleForSearch: boolean;
  isEligibleForPrediction: boolean;

  // validation
  errors: {
    activityType?: string;
    webpageURL?: string;
    userInfo?: string;       // duplicate-key error
    requiredKeys?: string;   // missing-key error
  };
}
```

A pure helper `composerStateToDefinition(state): ActivityDefinition | null` returns `null` when any error is set (which gates the "Become current" button per FR-010).
